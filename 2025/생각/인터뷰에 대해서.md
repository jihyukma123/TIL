# 생각

인터뷰 관련해서 영상을 보다가 면접관 경험이 있는 사람이 이렇게 말했음.

`I dont trust you`

이를 반대로 말하면

면접을 보는 내 입장에서는 나를 불신하는 면접관을 설득해야되는 자리임.

그래서 최대한 설명을 잘 해야되고, 필요하면 적극적으로 해명하고 나를 어필을 잘 해야되는 자리가 됨.

# 면접관은 '정답' 그 이상이 필요함

> I've gotta see what happens when you struggle.

면접관이 어떤 질문을 했을 때, 단순히 정답을 제시하는 것은 나를 파악하는데 별로 도움이 안됨.

왜냐하면, 생각의 과정을 볼 수 없기 때문에.

처음 보는 문제를 마주했을 때 어떻게 그 문제를 풀어가는지에 대해서 알고 싶은거임 -> 이를 통해서, 같이 일할 때 문제를 해결해나가는 방식을 엿볼 수 있기 때문에

그래서 인터뷰에서 Getting Stuck 하는 것은 마냥 나쁜 것이 아님. 오히려 면접관이 의도한 상황일 가능성이 있음. 이걸 어떻게 해결해나가는지

이런 상황에서 내가 어떤 식으로 문제를 해결하는 모습을 보여줄지, 무엇을 중요하게 생각하는 사람인지에 대해서 평소에 생각이 필요함.

# 릿코드와 같은 문제를 풀어보는 것과 실제 인터뷰 상황에서의 차이점

릿코드: 친절함. 전체 문제 컨텍스트와 샘플 데이터 제공

인터뷰: 문제 상황만 제시

왜 이런 차이가 발생하는지?

이런 상황이 주어졌을 때 면접관이 기대하는 것은, 지원자가 주어진 문제 상황을 해결하기 위해서 필요한 추가 정보를, 면접관과의 적절한 소통을 통해서 얻어내면서 문제를 해결해나가는지를 보고 싶은거임.

하다못해, 질문을 했어 -> 근데 질문을 내가 제대로 못알아듣거나 못들었어.-> 그러면 다시 한 번 말해달라고 하는것조차도 일부인거임.

이걸 면접이라고 생각하는것보다 그냥 회의실에서 같은 문제를 해결하고자 하는 동료 두 명이서 대화한다고 생각하는게 나을듯?

참고로, 질문 듣자마자 바로 코드치는 사람들 보면 그냥 바로 '안되겠구나'싶다고 함..허허

# 면접관은 나를 돕고 싶어함.

물어보면, 줄 수 있는 정보는 다 준다.

그리고 물어보라고 권장함. 문제를 해결하기 위해서 필요한 다양한 정보와 방향성을 잡아보자 함께.

# 자료구조, 알고리즘, 빅오 등등을 공부하는 이유?

어떤 문제를 해결하기 위해서.

근데 저런 것들을 활용해서 문제 어떻게 해결함?

크게 보면 문제 해결 방법들이고...동시에 문제를 해결할 수 있는 `도구`라고 생각해볼 수 있음. 그러면 못을 박아야한다라는 문제를 해결하는데는 망치라는 도구가 적합한 것처럼, 특정 문제를 적절하게 풀어낼 수 있는 알고리즘과 자료구조가 존재한다.

문제가 주어짐 -> 문제에 대한 정보를 수집함 -> 수집하는 과정에서 사용할만한 도구를 선택해나가는 과정이라고 볼 수 있을 것 같음.

그러니까,

문제가 주어짐 -> 문제를 해결하기 위해서 적절한 도구가 무엇인지 결정하기 위해서 필요한 정보수집 단계를 건너뛰고 바로 문제를 해결하기 시작함 이거는 좀 위험하지.

예를 들어, 널빤지를 벽에 붙여야 한다 라는 문제가 주어졌을 때, 바로 '못으로 벽에 고정시킨다'라는 문제 해결법을 답으로 인지하고 실행에 옮기는 거랑 비슷함.

it might work, but it also might not work.

이 문제는 추가 정보가 필요하다

- 널빤지가 벽에 어떻게 붙어야 하는지
- 널빤지가 못질을 했을 때 견딜만큼 튼튼한지? 못질했을 때 쪼개져버릴 가능성은 없는지?
- 벽에 붙여야하는 방향이 정해져있는지?

등등....이런 정보를 수집한 후에야 제대로 문제를 해결할 수 있음.

그치 문제를 파악하는게 중요하지.
