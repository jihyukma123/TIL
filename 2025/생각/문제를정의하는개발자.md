# [AI 시대, 개발자 커리어의 새로운 좌표](https://www.productengineer.info/community/articles/60098858-851c-49cc-8caf-41e17d283dc7)

커리어 고민에 대해서 느낀 바가 있어 글 내용과 함께 생각을 정리해봄.

AI와, 업계 표준 수준의 프레임워크를 활용해서 개발하는 시대에 개발자는 다 비슷해지는 것 같다...라는 말이 많이 공감되었음.

내가 일하는 환경에서 1년동안 경험했던 것도 비슷한 느낌이었다. React/FastAPI 기반으로 코드를 작성하는데 대부분의 코드는 Cursor에 내장된 LLM이 작성하고 나는 확인 후 수정요청하거나 confirm 하는 정도만 했음.

거의 대부분의 개발자들이 최근에 이런 방식으로 업무를 보고 있을 것 같음.

그러면 개발자는 이제 어디서 본인만의 엣지를 찾아야 하는걸까.

내가 가치있는 Worker라는 점을 어떻게 증명하는걸까...

## AI로 인한 개발 트렌드 변화를 정확하게 보자.

- Anthropic CEO: "3~6개월 내에 AI가 전체 코드의 90%를 작성하게 될 것이다. 12개월 후에는 사실상 모든 코드를 AI가 작성하는 세상이 올 수 있다."
- Google에서는 이미 신규 코드의 25% 이상이 AI로 생성되는 중
- Morgan Stanley 보고서: "AI가 코딩을 대체하는 것이 아니라, AI가 코딩의 병목을 다른 곳으로 옮기고 있다. 이제 병목은 ‘코드 리뷰’, ‘테스트’, ‘설계 결정’에 있다."

AI 이전에 개발자들의 핵심 업무 중 하나였던, 아이디어를 실제 코드로 "구현" 하는 작업의 희소성이 거의 사라짐.

"구현"의 희소성이 사라졌다는 표현이 진짜 정확한 것 같음.

"구현"의 희소성이 사라졌다는 말을 정확히 더 정의하긴 해야겠지만...

이 질문에 대해서 답이 안나오더라.

`주어진 요구사항을 잘 구현해왔고, 팀 안에서도 성실한 개발자였다` 정도가 딱 나한테도 들어맞는 표현인 것 같음.

# 기술 업계가 원하는 개발자상의 변화

Opcito Technologies의 2025 트렌드 리포트에서 언급된 내용:

`엔지니어들이 점점 더 아키텍트, 디자이너, 문제 해결자(problem solver)처럼 변하고 있다. 시스템 아키텍처 정의, 사용자 경험 설계 같은 고차원적 업무에 집중하게 된다.`

성공적인 SE가 가지고 있는 역량은...

- 제품 지향적 사고: 고객의 니즈와 비즈니스 결과를 기술적 결정에 연결할 수 있는 능력(나는 항상 이런 추상적인 단어의 나열이 도저히 와닿지가 않더라.)
- T자형 역량: 하나의 전문성을 깊게 가지면서, 여러 도메인을 넓게 이해하는 능력
- 시스템 관리 능력: 배포, 모니터링, 확장성을 코드만큼 잘 다루는 능력 (이거는 나도 느끼는건데, 근래에 프로젝트 하면 짧은 단위/적은 수의 사람으로 구성된 프로젝트가 많음. 그래서 CDS분들이 계신다 해도 개별 개발자들이 배포/모니터링 관련 역량이 있고 없고의 차이가 클 것으로 예상됨)

# SE -> Product Engineer라는 역할로의 진화?

무엇을 만들 수 있느냐? 보다, 어떤 문제를 선택하고 풀 수 있느냐가 더 중요해질 것.

예전처럼, 남들이 만들어준 기획/디자인에 대해서 잘 구현하는 것에 집중하는건 이제 시대가 달라졌다는 생각...(이전 직장에서의 일하던 방식이었는데, 이제는 거기서 벗어날때가 되었다.)

생각해보니 근래에 가장 좀 충격적인? 구현 관련해서 진짜 내가 할게 없구나라고 느낀 부분이 이거였음. 프로젝트에서 분할화면이 필요하다고 해서, 프롬프트를 통해서 해달라고 했더니 금방 구현해줬는데, 아무것도 수정할 필요가 없이 내가 원하는 느낌과 디자인으로 바로 구현됨. 분명히 직접 하려고 하면 꽤 오래 걸렸을텐데....

그러면 새로운 시대의 SE, 즉 Product Engineer는 어떤 사람이지?

- 문제를 관찰하고
- 이게 기술로 풀 가치가 있는지 판단하고
- 구조를 설계한 다음
- 기술을 도구로 사용하는 사람

저번 주인가? 프로젝트 지원 들어가서 싱크 회의에 참석했는데, 선배 개발자가 프로젝트 전반에 대해서 제시하는 의견을 듣고 충격을 받았던게 생각남.

- 고객이 제시한, 원하는 기능에 대해서, 현실적으로 우리가 어렵게 기술로 풀지 않아야하는 문제라고 생각하는 부분을 지적함. -> 나는 그냥 아우 빡시겠네...어떻게 풀어내지? 라는 생각을 혼자 하고 있었는데, 애초에 이게 우리가 풀어야할 문제인지? 아니면 다른 방식으로 풀어냈을 때 더 잘 풀어볼 여지가 있는 문제인지에 대해서 더 앞단에서 고민을 하는 모습을 보고 배웠음. -> 이 경험은 따로 또 정리를 하는게 좋겠다.

# 구현의 희소성이 사라졌다는 의미는?

1. 과거에 “구현”이 왜 희소했는가

AI 이전의 구현은 다음 능력들의 결합 상품이었습니다.

- 요구사항을 코드 구조로 분해하는 능력
- 언어/프레임워크 문법 숙련도
- 에러를 디버깅하며 끝까지 완성하는 인내력
- 반복적인 boilerplate를 빠르게 찍어내는 손 기술

이 중 상당 부분은: 시간이 오래 걸리고 숙련자만 안정적으로 수행 가능했기 때문에 👉 *“아이디어 → 실제 동작하는 코드”*로 변환하는 능력 자체가 희소했고, 그 자체로 가치가 있었습니다.

2. “구현의 희소성이 사라졌다”의 정확한 의미

핵심 정의 -> 아이디어를 코드 형태로 변환하는 행위 자체가 더 이상 차별적 가치를 만들지 못한다

조금 더 정확히 말하면: 코드를 “작성하는 행위”의 한계 비용이 급격히 0에 수렴

“어떻게 쓰느냐”보다 **“무엇을 쓰게 할 것인가”**가 중요해짐. 즉, 구현은 여전히 필요하지만 구현을 할 수 있느냐 없느냐가 실력의 경계선이 되지 않음

3. 구현이 사라진 게 아니라, “자동화된 하위 작업”이 됐다

중요한 오해 하나를 짚어야 합니다.

- ❌ 구현이 필요 없어졌다
  -> ⭕ 구현이 의사결정의 결과물이 됐다

AI가 대체한 것은:

- CRUD 코드
- API wiring
- 반복적인 패턴 구현
- 문서화 가능한 표준 로직

이들은 이제: “사고의 대상”이 아니라 **“사고 후에 자동으로 생성되는 산출물”**이 됨

👉 구현은 판단이 끝난 뒤 버튼을 누르면 나오는 출력물에 가까워졌습니다.

4. 그래서 Morgan Stanley가 말한 “병목 이동”이 핵심

AI가 구현을 가속하면서 드러난 진짜 병목은 다음입니다.

4-1. 설계 결정

- 무엇을 abstraction으로 둘 것인가?
- 어디까지 일반화할 것인가?
- 미래 변경 가능성을 어디까지 고려할 것인가?

👉 이건 언어가 아니라 문제 구조에 대한 이해 영역

4-2. 코드 리뷰

- 이 코드가 “의도한 문제”를 정확히 풀고 있는가?
- 장기적으로 유지 가능한 구조인가?
- 성능/보안/확장성의 위험은 없는가?

👉 AI는 생성은 잘하지만
👉 “의미 평가”와 “맥락 판단”은 여전히 약함

4-3. 테스트

- 무엇을 테스트해야 하는가?
- 어떤 실패가 “중요한 실패”인가?
- 테스트 비용 대비 리스크는 적절한가?

👉 테스트는 코드가 아니라 제품 가설에 대한 검증 설계

5. “구현의 희소성 상실”을 한 문장으로 요약하면

이제 구현 능력은 ‘입장권’이지 ‘차별화 요소’가 아니다

예전:
“이걸 구현할 수 있는 사람” = 가치

지금:
“무엇을 구현해야 하는지 올바르게 결정하고,
결과물을 평가·수정·통합할 수 있는 사람” = 가치

6. 개발자의 역할은 이렇게 재정의된다

| 과거 중심 | AI 이후 중심 |
| --------- | ------------ |
| 코드 작성 | 문제 정의    |
| 문법 숙련 | 시스템 사고  |
| 구현 속도 | 설계 판단    |
| 디버깅    | 리스크 관리  |
| 기능 완성 | 제품 품질    |

7. 그래서 이 변화의 진짜 함의는?

주니어-시니어 격차는 줄지 않고, 오히려 커질 가능성

“코딩 잘함”보다
“생각을 잘 구조화함”이 압도적으로 중요

개발자는 점점:

코더 → 엔지니어 → 의사결정자로 이동
