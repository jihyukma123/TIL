`객체지향`이 뭘까.

포프 아저씨가 이름만 계속 바뀔 모든 것이 객체지향이라고 해서 관심이 생겼음.

내가 요즘 생각하는 '본질'에 가까운 무언가를 배울 수 있지 않을까 싶기도 하고.

결국 쪼갤거 잘 쪼개고, 합칠 건 잘 합쳐서 프로그램 잘 만들면 되는건데 너무 복잡하게 생각하는 듯.

잘 쪼개려면 뭘 알아야 하는지, 잘 합치려면 뭘 알아야 하는지에 대해서 생각해보면서 읽어보려고 함.

# 객체지향 4단계

1단계: 클래스가 아니라 객체를 바라보는 것
2단계: 객체를 독립적인 존재가 아니라, 기능을 구현하기 위해 협력하는 공동체의 존재로 바라보는 것
3단계: 협력에 참여하는 객체들에게 적절한 역할과 책임을 부여하는 것
4단계: 내가 사용하는 프로그래밍 언어라는 틀에 1~3단계를 잘 담아내는 것..

객체지향 언어를 사용하는 개발자를 대상으로 썼다는데...JavaScript는 그런 언어일까..?

JavaScript가 객체지향 언어인가 아닌가는 중요한 게 아닌 것 같음.

그냥 독립적인 단위로 개념들을 책임을 적절하게 부여해서 묶고, 서로 잘 소통하고 하면 되는거지.

# 1장: 협력하는 객체들의 공동체

> 전체는 부분의 합보다 크다.

역할, 책임, 협력.

실세계를 모방한다면 이걸 모방하는 것이라는 걸 표현하고 싶었던 것 같다.

실세계의 무언가가 객체로 치환되는 것보다, 실세계에서 어떤 작업이 이루어지는 방식은, 각자 맡은 역할과 책임을 협력하는 주체들이 협력해서 일련의 작업을 해서 결과물이 만들어지는거니까.

책에서 제시한 커피 공화국 예시를 보면,

- 손님 -> 캐셔 -> 바리스타 -> 손님으로 이어지는 과정이 계속 반복되면서 '커피 주문'이라는 이벤트가 처리되는 것.
- 손님의 역할과 책임은 무엇인가?
- 캐셔의 역할과 책임은 무엇인가?
- 바리스타의 역할과 책임은 무엇인가?

'협력'이라는 단계는 이 '역할' 및 '책임'이라는 것과는 성질이 좀 다른 걸로 생각됨. 왜냐하면 '역할' 및 '책임'은 좀 더 self-contained인데 협력은 2개 이상의 주체가 관여되어야 한다고 생각해서.

개발을 할 때, 설계를 할 때, 내가 작성하는 코드가 구성하는 시스템 내에서 '역할'과 '책임' 관점으로 바라보는 건 도움이 되는 시선일듯?

지금까지 그렇게 생각해본적이 있나? 자연스럽게 구조적으로 나눠서 생각하는 정도였지 명확하게 역할과 책임에 대해서 생각해본 적은 없었던 것 같음.

여기서 '협력'은 '커피 주문'임. 개별 역할/책임 주체들이 커피 주문이라는 협력 관계를 가지는 것이라고 나는 이해되는데.

역할과 책임이 좀 헷갈리는 부분은, 둘이 서로 뭐가 다른지 잘 모르겠다는 점.

역할 - 협력에 참여하는 객체에 대한 페르소나, 관련성 높은 책임의 집합

하나의 역할에 여러 비슷한 책임들이 소속되는 느낌인 건 이해를 했음.

예를 들어서, LG CNS 에서 나는 '풀스택 개발자'라는 역할이 있고, 이에 따라 다음의 책임이 요구됨

- React, Spring Boot 기반 앱 개발
- AI Agent 개발
  등등...

## 협력의 기본은 요청과 응답 + chain of that

사람들끼리 일하는 방식을 생각해보면, 요청과 응답으로 협력이 진행된다. (여러 사람이 involve된 협력이라면 연쇄적인 요청과 response의 chain으로 진행되겠지.)

객체지향에서 객체 간 협력도 이런 관점에서 이해하면 객체들끼리 요청 응답을 주고 받는거임.

협력이 성공하려면, 그 협력을 진행하는데 들어오는 요청과 응답을 처리하는 객체들이 각각 요청을 잘, 성실하게 수행해야됨.

간단하게 말하면, 여러 명이 같이 일을 하는데 각자 자기 할 일을 잘 해야지 전체 일이 잘 돌아간다는 소리임.

약간 좀 그런 관점이 보이기도 하네. 각각 작은 단위의 역할주체들에게 적절한 역할과 책임을 부여하고, 부여받은 일을 잘 하는 것이 전체가 잘 돌아가는 기반이다 같은......뭐라고 해야되지 각자 잘해야지 모두 잘된다?

갑자기 이런 생각이 들었음.

지난 프로젝트에서 내가 다이얼로그 컴포넌트를 useDialog라는 custom hook으로 뺐었는데, 그 때는 그냥 '재사용성'이랑...'캡슐화'? UI만 만들려다가, 아니 근데 어차피 이 Dialog가 show/hide되는 여부를 관리하는 변수도 항상 필요한데 그냥 묶어버리면 안되나? 그리고 어차피 필요한 건 이 상태를 관리하는게 아닌데 왜냐하면 상태는 true/false만 가지면 되는 toggle인데 open/close라는 컨트롤 로직만 밖으로 빼면 되지 않나 하는 생각이 들어서.

근데 지금까지 읽은 내용을 토대로 해당 과업? 작업을 생각해보면, 우선 해당 컴포넌트의 '역할', '책임' 그리고 해당 컴포넌트가 '협력'이라는 관점에서 어떻게 요청을 받고 처리하는지에 대해서 생각해볼 수 있는 것 같음.

역할이 뭐지? UI를 보여주는 것? 이라고 하면 state를 넣어도 되고 안넣어도 됨.

근데 역할을 'Dialog를 보여주기 위한 UI와 작업을 처리`라고 하면 state를 포함시키는 것이 맞음.

책임은 여러가지지

- UI 표시
- Dialog 표시/hide
- 상태관리

충분히 협력적일려면? 쉽게 사용할 수 있는게 중요하지 않을까. '보여줘'라는 요청을 잘 처리하면 되는거지.

## 핵심은 객체

기저에 깔려있는, 객체지향이 돌아가는 flow라고 해야되나? 기저 원리라고 해야하나 무튼 뭔가 그 근본이 되는 구조는

역할, 책임, 협력임.

근데 역할을 맡아서 책임을 수행하고 협력하는 주체 자체는, `객체`이다.

`객체`지향이잖아 말그대로.

`객체`는 협력하는 공동체의 일원이라는 측면에서 갖춰야할 덕목이 2가지 있음. (참고로 언제나, ALWAYS 협력체의 일원인 객체만 생각하면 됨. 왜? 프로그래밍이라는 관점에서 보고 있는데 현실적으로 단일 객체로 구성된 실제 프로그램은 있기가 힘드니까. 모든 프로그램은 여러 객체들로 구성될 수 밖에 없으니까, 하나의 프로그램이 수행하는 큰 기능을 달성하기 위한 보다 작은 책임들을 수행하는 특정 역할 객체들의 협력이라는 관점에서 보는 것이 필요하다.)

덕목 2가지는..

1. 충분히 `협력적`이어야 한다.

말 그대로, 다른 객체들이랑 협력할 의지가 있는 친구여야 한다는 뜻임. 이게 협력을 할 수 있는 구조여야 한다 이렇게 이해되기도 하는데, 이거는 너무 실제 구현 관점에서의 서술 방식인 것 같음. 객체가 가져야 하는 구조, 다른 객체와 협력할 수 있는 구조 이런걸 고려하기 전에 애초에 객체 자체에 대해서 내가 생각할 때, 이 친구는 협력자이고 잘 협력하려면 뭐가 필요하지 같은, 조금 더 사람에 가까운? 무튼 뭔가 코드보다는 더 관계에 집중한 그런 느낌에서 먼저 생각을 해봐야 하는 것 같음.

협력적 -> 요청에 응답한다는거고, 응답할지 말지, 어떻게 응답할지는 객체의 choice.

2. 충분히 `자율적`이어야 한다.

자율적이라는 것은, 본인의 행동을 스스로 결정하고 책임진다는 것.(스스로 결정 + 그에 대한 책임을 지는 것.)

사실 이 '자율적'이라는 개념이 여전히 좀 와닿지 않기는 한다.

결국 코드를 짜는대로 행동하는데 뭐가 자율적이라는거지? 이러면 너무 인간 중심 사고인가 흠.

객체의 자율성이란, 내가 이해한건 이거임.

- 주어진 역할, 책임은 알아서 함. 이거는 자기가 알아서 하는거고 남이 건드리거나 알아야 하는 내용이 아님.
- 본인과 협력할 수 있는 방법을 자기 방식대로 제공함.

그러면 외부에서 봤을 때 이 객체는,

- 어떻게 하는지는 모르겠지만
- 이 친구가 요청하라는 형식에 맞춰서 요청하면
- 나한테 응답을 줌

how? i dont know. the `outside` of the object does not know, and should not know 'how' the object handles the request. but the outside should know 'what' the object does.

바리스타는 커피를 알아서 만드는 자율성이 있음. 커피를 만들어달라고 컵을 주기만 하면, 찌는 굽든 그건 그 사람 선택임. 그런 자율성을 의미하는 것 같음. 일을 어떻게 처리하건 정해진 일을 잘 처리하면 됨 안에서 어떻게 일을 처리하는지는 바깥하고 상관이 없음.

객체에 data와 data를 다루는 Method가 같이 묶이는 건 이 자율성을 위해서라고 이해함.

자율적으로 행동하려면, 본인의 책임 범위 내에 있는 일을 수행하기 위해서 필요한 요소들을 다 가지고 있어야지.

커피 제조법이라는 데이터와, 커피 제조법대로 커피를 만든다 라는 액션을 둘 다 할 수 있어야 함 바리스타가.

그래야지 '커피 제조'라는 역할을 온전히, 자율적으로 수행할 수 있음.

## 객체의 유일한 소통방법: 메시지

객체들은 오로지 message로만 소통한다고 함.

이게 무슨 소리지?

메시지와, 메세지를 전달하는 방법..?

메세지를 전달하는 방법이 메서드인가 라고 나는 생각했는데, 메서드는 메세지를 전달받는 방법이라고 함.

...메세지와 메서드를 분리해서 생각한다라...

메세지를 전달받았을 때, 메세지에 대응되는 특정 메서드가 실행되는 구조라는 것.

(근데 이상한게, 메서드를 실행 시점에 선택할수가 있나 받는 쪽에서?? 애초에 호출하는 쪽에서 여러 메서드 중에서 한 가지를 선택해서 호출하는거 아니야? -> 이에 대해서 공부하고 글을 썼음. [객체지향과 절차지향의 차이점 중 한가지 이해](../2026/2/객체지향과%20절차지향의%20차이점%20중%20한가지%20이해.md))

참고로 OOP를 창시한 Alan Kay는 이렇게 말했다고 함. 'OOP의 핵심은 메시지다'

메세지와 메서드가 분리된다. -> 이거는 메세지가 어떤 세부 로직이 실행될지 결정하지 않는다 라는 의미로 이해함.

메세지는 전달되고, 끝. 메서드는 메세지에 의존하는게 아니라, 내부적으로 결정됨. 왜? 메서드는 '메세지를 처리하는 방법'이라고 했으니까.

객체가 수신한 메세지를 처리하는 방법이 메서드니까, 메세지가 메서드를 결정하지 않지. method를 호출되는 함수 그 자체로 보는게 아니라, 그 함수가 호출되었을 때 실제로 실행되는 로직, 커피를 만드는 로직으로 보면, 커피를 만들어라는 메세지가 어떻게 커피를 만들지를 결정하지 않는다는 거고, 그 관점에서 '커피 만들어줘'랑, '핸드드립으로 내린다'는 완전히 분리되어 있음.

이렇게 분리를 시켜두니까, 객체의 자율성이 높아짐. -> 어떻게 만드는지는 오로지 나의 소관이니까.

코드 관점에서 보면 이해가 어려울 수 있는데 관계 관점으로 보니까 이해가 되는 부분들이 있네 확실히

둘다 봐야하는 것 같음.

## 클래스에 집착하지말고 객체, 그리고 객체의 역할/책임 및 객체 간 협력에 집중하라.

너무 클래스에 집착하는 경우가 많다고 함 근데 그거는 굉장히 좁은 의미의 객체지향일 뿐.

객체라는 것은 클래스와 동의어가 아니다.

객체라는 것은 사실 클래스도 아니고, 어떤 프로그래밍 언어의 그 무엇도 아니다 그냥... 객체임.

어떤 역할을 가지고 해당 역할에 속하는 몇 가지 책임을 수행하며 그 역할을 필요로 하는 다른 객체가 요청 메세지를 보냈을 때 그걸 자율적으로 처리해서 response를 주는 형식으로 협력하는 주체? 같은 느낌으로 이해됨.

객체를 나타내는 클래스를 어떻게 구성할까보다는, 협력 구조와 책임을 식별하는 것이 훨씬 더 중요하다는 점.

서로 메세지를 주고 받는 객체들의 관점에서 OOP를 이해하려고 노력해야 된다.

여러 클래스들 간 관계를 보는게 아니라.

그런 관점에서 커피샵 예시 참 적절한 것 같음.

관계로만 보면 그 소통, 협력하는 과정이 이해가 잘 안될 수 있지 않을까?

손님 객체, 캐셔 객체, 바리스타 객체. 클래스만 보면 얘네가 서로 어떻게 소통한다는 건지 바로 직관적이지 않을 수 있음.

근데 큰 목표 -> 그 목표를 달성하기 위해서 각자 수행하는 역할과, 그 역할을 수행하는 프로세스를 보면? 협력을 하는 방법을 보면?

손님 -> 캐셔 -> 바리스타 -> 손님 으로 이어지는 흐름을 볼 수 있고,

손님은 주문하는 역할
캐셔는 주문을 받고, 바리스타에게 커피 주문 전달하는 역할
바리스타는 커피를 만들고 다시 손님에게 전달하는 역할

그리고 협력은 객체에서 발생하는게 아니라, 객체 간 소통에서 발생함.

손님 -> 캐셔 -> 바리스타 -> 손님 이 flow에서 화살표가 소통이고 협력임.

request-response로 구성된, 그리고 request-response를 처리하는 방법은 message를 전달하는거고, 메세지는 어떤 코드를 실행할지가 아니라 무엇을 수행할지에 대한 내용임.
