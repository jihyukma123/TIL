# 이해가 안되는 부분..

`객체지향의 사실과 오해`라는 책을 읽어보는 중인데, 거기에 나오는 다음 내용이 이해가 안돼서 생각해보고, AI와 대화하면서 이해해보고자 한다.

객체지향에서, 객체들이 서로 소통하는 수단은 단 한가지, 메세지임.

메시지를 전송해서 요청하고, 수신해서 요청을 받고.

객체가 수신한 메세지를 처리하는 방법이 `method`이다. 어떤 객체에게 메세지를 전송하면 메세지에 대응되는 메서드가 실행됨.

근데 여기서, `메세지를 수신한 객체가 실행 시간에 메서드를 선택할 수 있다는 점은 다른 프로그래밍 언어와 객체지향 프로그래밍 언어를 구분 짓는 핵심 특징 중 하나다`라고 하는데...

이게 정확히 무슨 말인지 이해가 잘 안된다.

## 절차지향은 컴파일 타임에 결정된다는 의미.

절차지향 언어는 컴파일 시점에 프로시저 호출에 대한 실행 코드를 결정한다는데, 이게 무슨 뜻일까.

```C
add(1,2) # 이걸 봤을 때, add라는 이름이 가리키는 코드가 정확히 무엇인지(어디인지) 컴파일 시점에 확정됨. add는 무조건 그 단일 함수이고 다른 함수가 실행될 가능성이 없음.
```

그러면 궁금한게, a.speak()을 구현하고 a에 어떤 객체가 바인딩되냐에 따라서 동적으로 호출하는 그런 구조의 코드를 C같은 절차지향 언어에서는 아예 구현할 수가 없나? 궁금해서 물어봄.

답은...`하려면 할 수는 있다. 근데 문제는 언어가 native하게 지원하지 않는 기능이기 때문에 너가 직접 한땀한땀 만들어서 써야한다`

`절차지향 언어는 컴파일 시점에 실행 코드가 결정된다`라는 것은 엄밀히 말하면 언어의 기본 함수 호출 메커니즘이 컴파일 타임에 바인딩 되는 것이라는 소리임.

언어가 지원하는 프로그래밍 컨셉에 객체지향의 다형성이 고려되어 있지 않다는 소리로 이해함

## 그러면 객체지향 언어는?

객체지향 언어는, 실행되는 함수가 정확히 어떤 함수인지 컴파일 하는 시점이 아니라 런타임에 동적으로 결정될 수 있는 구조 자체가 언어 차원에서 native하게 지원됨

```java
class Animal {
    void speak() { System.out.println("..."); }
}

class Dog extends Animal {
    void speak() { System.out.println("멍멍"); }
}

class Cat extends Animal {
    void speak() { System.out.println("야옹"); }
}

public class Main {
    public static void main(String[] args) {
        Animal a;

        if (args[0].equals("dog")) {
            a = new Dog();   // a가 실제로 Dog 객체를 가리킴
        } else {
            a = new Cat();   // a가 실제로 Cat 객체를 가리킴
        }

        a.speak();  // 여기서 뭐가 호출될까?
    }
}
```

컴파일러가 봤을 때, a는 이미 Animal 타입이지만, 문제는 Animal 타입을 extend하는 여러 객체 중 어떤 친구의 메서드가 실행될지는 모름.

그래서 런타임에 어떤 값이 프로그램 실행 과정에서 전달되느냐에 따라서, 작성된 코드는 하나인데(main 함수 안에서는) 동작은 여러 가지로 달라질 수 있음.

## 수신자가 실행 메서드를 선택한다...?

오케이 객체지향 프로그래밍 언어가 native하게 동적으로 실행 시점에 다른 로직이 동일한 코드에서 실행되도록 구현되어 있다는 건 이해함.

그리고 이게 확장성/변경에 더 유용한 측면이 있다는 것도 이해했어.

근데 그래도 이해가 안되는건

이 구조가 메세지 수신자가 실행될 메서드를 결정하는 형태야? 그래도 어떤 객체의 speak함수가 실행될지는 여전히 메세지를 호출하는 사람이 결정하는거아니야?

어떻게 '받는 사람'이 실행될 메서드를 결정한다는 건지 이해가 잘 안됨.

일단 이 문제에 대해서 한 가지 정리하고 넘어가야 하는 점은, 호출자와 수신자를 잘 구분해야 된다는 점.

호출자는 수신자 안에서 어떤 로직이 실행되는지 몰라도 된다라고 이해하는 게 맞는 것 같음.

```javascript
// Java를 잘 몰라서 JavaScript로 코드를 작성함.
class Barista {
  makeCoffee() {
    console.log("그냥 커피");
  }
}

class Minsu extends Barista {
  makeCoffee() {
    console.log("핸드드립으로 내림");
  }
}

class Jiyoung extends Barista {
  makeCoffee() {
    console.log("머신으로 내림");
  }
}

class Taeho extends Barista {
  makeCoffee() {
    console.log("콜드브루로 내림");
  }
}

class Cacher {
  deliverCoffeeOrder(worker) {
    worker.makeCoffee();
    console.log("커피 제공 완료");
  }
}

// --- 실행 ---
const staff = [new Minsu(), new Jiyoung(), new Taeho()];
const todayWorker = staff[Math.floor(Math.random() * 3)];

// 처음에 내가 헷갈렸던 부분은, 이 cacher.deliverCoffeeOrder가 실행되는 이 area를 호출자라고 생각했음
// 그래서 호출자가 todayWorker가 누군지 알고 호출하는데 뭔 수신자가 결정한다는거여 라고 생각했는데,
// 호출자는 cacher임. cacher는 todayWorker가 누구인지 모름.

// 그냥 Order를 받아서 todayWorker가 누구인지 모르겠는데 호출하는거야
// worker를 얘가 호출한다 이런 관점보다.

// worker에게 '커피 만들어줘'라는 메세지를 전달하는거지.

// 그러면 worker가 누구인지는, 그래서 어떻게 커피가 만들어질지는 worker의 선택. 즉 수신자의 선택이다.
// 호출자가 선택하는게 아님 어떻게 커피가 만들어지는지, 즉 요청에 대해서 response가 어떻게 처리되어서 나오는지
// 이거는 receiver가 결정하는 문제. 머신으로 할지 콜드브루할지 핸드드립 할지는 worker가 결정하는 문제이지, sender가 결정하는 문제가 아니라는 점. 절차지향이라면, 머신으로해줘/핸드드립해줘 등등 했을텐데 객체지향에서는, '커피 만들어줘' 하면 어떻게 할지는 메세지 수신 객체의 영역임. 즉 makeCoffee라는 함수가 어떻게 커피를 만들지 안에서 결정됨.

const cacher = new Cacher();

cacher.deliverCoffeeOrder(todayWorker);
```

Aha!!! 이것저것 좀 하면서 이해가 됐다.

수신자가 결정한다는 건 코드 관점에서의 서술이라기 보다는, 관계의 관점에서

메세지를 전달하는 사람은 how를 모르고 what만 안다는 것. how는 receiver의 책임/역할 범위에 있는 일이라는 것.

잘 분리된 책임/역할 구조에서의 협력이라면 캐셔가 커피를 만드는 방법을 알 필요가 없고, 바리스타가 '알아서' 할 문제임. 그런 관점에서 '결정의 주체가 수신자다'라는 것을 이해했음. 그리고 반대로 바리스타도 캐셔가 주문을 현금으로 받았는지...카드로 받았는지는 아예 알 필요가 없음

그냥 무슨 커피를 만들어야할지만 알면 됨...(그런 관점에서는 makeCoffee가 아니라, makeAmericano, makeLatte 이런 식으로 구현되어야겠지 무튼)
