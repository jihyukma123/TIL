# 처음으로 TDD를 해보기.

TDD 재밌다!

생각보다 장점을 바로 느낄 수 있었음.

1. 함수가 수행하는 목적에 대해서 생각하게 됨
2. 로직에 대해서 미리 생각하게 됨
3. 새롭게 추가한 로직 때문에 기존 전제가 어긋나지 않는지 안심하고 코드를 작성할 수 있음.
4. 미리 테스트를 작성하고 테스트를 수행하는 방식으로 코드를 짜니까, 어디서 어긋낫는지 좀 더 직관적으로 알 수 있음(console.log를 찍는거보다 이게 더 낫다고 생각되기도 함 어떨때는?)

어떤 장점이 있는지 몰랐는데 바로 와닿네

약간, 무턱대고 코드부터 치게 되는 것에 대해서 그러지 말아야겠다고 생각이 들었던 순간들이 있었고, 어느 순간부터는 뭘 구현해야되는지 생각을 먼저 많이 했던 것 같음.

근데 TDD는, 애초에 코드부터 무턱대고 치는걸 방지해주는 방법론이긴 하다.

확실히 먼저 생각을 해야됨.

test부터 쓴다. -> 무엇에 대해서 테스트를 써야하지를 생각해야하고, 그러면 당연히 뭘 만들어야하는지에 대해서 생각하게 됨.

물론 결국에는 더 큰 범위에서, 뭘 구현할지에 대해서 먼저 생각하고 큰 그림을 이해하고 코드를 작성하자.

즉 코드는 생각의 표현일 뿐이라는 점에서는 설계를 먼저하고 코드를 쓰고 테스트를 쓰건, 설계를 하고 테스트를 쓰고 코드를 쓰건 큰 차이는 없을 것 같음.

중요한건 계획을 세우고, 의식적으로 코드를 써야한다는 것.

손이 가는대로 쓰는게 아니고, 구조와 큰 그림 속에서의 의미와 경계 같은 것들을 이해하는 노력이 필요하다.

결국에는 코어한 부분을 더 잘 이해/체득하고 있는 사람이 코딩에서도, AI assited coding에서도 유리하겠넹.

어디를 나눠야할지, 의존 방향이어디로 가야하는지, 경계가 어디인지, 레이어가 어디인지 등등...

테스트를 나누는 기준도 거기에 달려있기도 하고.

AI한테도 일을 시킬 떄 제일 중요한거, 컨텍스트 엔지니어링이고 harness고 뭐고 간에 내가 느끼기에는 AI의 성능이 이미 어느정도는 한계에 왔음.

얘한테 알아서 잘 해줘 하면 잘 안돼 이거는 그냥 확률적으로 계속 틀어질 수 밖에 없는 문제야 10번 시켰을 때 웬만큼 일의 크기가 커지면 틀어짐.

그래서 틀어지는 걸 방지하려고 자꾸 노력하는 단계인 것 같음. 그리고 모델들도 자꾸 이런 작업을 잘 할 수 있는 방식으로 진화하고 있고.

그래서 일을 잘 시켜야하고 잘 시키려면 구조적 사고, 설계, 어디서 나누고 합치고 의존하고 쪼개고 테스트하고 이런 걸 알아야 한다.

모듈화, 응집도 높이고 결합도 낮추고, 의존성 잘 제어하고 등등 이런거 배우려면 어떻게 해야되지? 흠흠흠...
