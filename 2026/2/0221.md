# 코드의 완성도 != 프로덕트 완성도의 발전일까.

harness engineering 이라는 OpenAI 블로그 글을 이해하려고 parse, don't validate 같은 원칙을 같이 알아보다가 갑자기 이런 생각이 들었다.

해당 부분을 읽다가 보면 다음과 같은 내용이 나온다.

`By enforcing invariants, not micromanaging implementations, we let agents ship fast without undermining the foundation.`

번역하면, "구현 방식을 일일이 간섭(마이크로매니징)하는 대신 핵심 규칙(불변량)을 강제함으로써, 시스템의 근간을 해치지 않으면서도 에이전트가 빠른 속도로 결과물을 낼 수 있게 합니다." 라는 의미라고 함.

여기서 내가 든 생각은, 'not micromanaging'이라는 걸 나는 '코드를 일일이 어떻게 짜야할지에는 전혀 관여하지 않는다'로 이해를 했다.

그리고 이런 방향성은 최근에 내가 여러 개발 아티클들을 접하면서 느꼈던, '코드가 중요한게 아니다. 코드가 창출하는 가치, 해결하는 문제, 고객에게 전달되는 가치, 비즈니스 value 가 중요하다' 라는 메세지와 어느정도 일맥상통하는 부분이 있다고 느껴졌다.

2022년 쯤 개발을 시작할 때만 해도 이런 느낌보다는 개발자들은 조금 더 코드 그 자체에 대해서 더 논의를 많이 했던 느낌인데(개인적인 느낌이라 틀릴 수 있음.) 최근으로 오면서 이런 메세지가 더 많아진다고 생각을 많이 했었다.

나도 많이 고민을 했었고....맞는 말이긴 하니까. 코드를 잘 짜고, 좋은 프로그래밍 원칙이나 패턴을 적용해서 코드베이스 자체의 완성도를 추구하는 것이 꼭 그 코드가 만들어낸 프로덕트의 완성도를 의미하지는 않으니 말이다.
(스타트업에서 일하면서도 느꼈던 것 같다. 나름대로 내가 아는 범위 내에서 좋은 코드를 짤려고 많이 노력했던 기능이 출시 후에 유저들의 반응이 없는 경험을 했음)

근데 AI를 활용해서 코드를 짜는 시대에는 이런 경향성이 더 심해지지 않을까 하는 생각이 들었다.

신경쓰는 사람들은 신경 쓰겠지만, AI가 어떤 코드를 생성하건 잘 돌아가면 되는거 아닐까?

유지보수성이고 뭐고, 일단 엔드 유저 입장에서 원하는 프로덕트를 빠르게 deliver할 수 있으면 그 프로덕트 내부, black box를 구성하는 코드가 react건, vue 건 그런건 적어도 '개발자들이 좋아하는 코드 관점에서의 완성도'는 크게 중요하지 않은 시대가 와버린 것 같다.

컴포넌트 구조, view와 logic의 분리 등등...이런 것들도 의미가 있는걸까? 적어도 사람이 코드를 더 잘 이해하기 위해서 필요했던, 오로지 그 목적 하나만을 위해서 필요했던 구조는 만약에 AI Agent가 더 효율적으로 일하는데는 딱히 도움이되지 않는다면 더이상 가치가 없는 구조가 아닌가 하는 생각이 든다.

나만해도 근래에 이런 저런 코드를 짜면서 정확하게 한줄한줄 implementation을 신경쓰지 않는다. 그래도 잘 돌아가고...React 코드를 정확히 뭘 사용했는지 같은건 크게 중요하지 않은 느낌이랄까.(그래도 React를 쓰는게 통계적으로 효율적이긴 하겠지만)

이런 관점에서 보면, AI-assisted engineering은 더욱더 코드의 완성도가 프로덕트 완성도가 아니고, 개발자는 코드만 신경쓰는 존재가 아니다라는 담론을 그대로 이어받아서 더욱더 확장시키는, 혹은 심화시키는 방향성인 것 같다.

이제는 진짜 뭘 만드는지가 더 중요하지 어떻게 만드는지 적어도 세부적인 코드 한줄한줄에서의 의미와 관리는 중요하지 않게 된 것 같다.

근데도 알긴 알아야할까? 이게 참 어려운 부분이다 선택의 문제겠지. 나는 뭘 선택해야 할까.
