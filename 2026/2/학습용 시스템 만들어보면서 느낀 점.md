# AI 보조 쇼핑몰 엔지니어링 훈련 시스템 기초 구조 만들면서 느낀 점.

AI를 활용해서, 백엔드/풀스택 기반 완성된 앱을 만들어본 경험이 부족한 문제를 해결하기 위한 Agent를 만들어봤음.

커리큘럼대로 학습/복습을 지원하면서 멘토링을 하는 역할을 가진 agent 임.

- AI한테 문제를 설명하고, 이것저것 참고해서 학습을 도와주는 Agent를 만들어보라고 하니까 꽤 잘 만들었음
  - 장점: 내가 생각치 못한 부분을 챙겨준다. 예를 들어서, 학습/복습에 대해서 어떻게 학습하는게 과학적으로 효과적인지 논문을 활용해서 AI한테 지시하는 프롬프트를 작성해줌
  - 단점: 너무 과하게 설계함. 무슨 설계 폴더만 8개....

그래서 AI하고 대화하면서 설계하는 과정을 통해서 최종본을 만들었음.

아니 근데 다듬는 과정이 너무 오래걸리는거야...그러면서 든 생각이

- 처음부터 내가 명확하게 설계하고 지시하는게 나았을까? 였음.

이런 생각이 들었음.

- 자유도를 주면, 내가 생각치 못한 부분을 제안하기도 하지만, 내가 목적을 뾰족하게 달성하는 결과물이 바로 나올 가능성은 현저하게 떨어짐
- 너무 자유도를 주면, 만들어준걸 이해하고 수정하는데 시간이 오래걸림
  - 내가 AI가 만들어준 결과물을 이해하는데 걸리는 시간과 리소스, 즉 `human time and energy`가 희소한 리소스인데, 이게 너무 많이 소모됨.

AI가 만든걸 이해하는데 오래 걸린 이유가 뭘까?

- 내용 이해보다, 폴더를 8개 만들어줬는데 각 폴더와 그 안에 있는 파일들의 역할이나 관계 등 큰 그림/구조를 이해하는데 시간이 오래걸림.
- 세부 파일의 내용을 이해하는건 큰 병목이 아니었음.

AI가 뭔가를 생성하는 것보다, AI가 만든걸 이해하고 검수하고 수정하는게 더 오래걸리고 인간 리소스가 많이 들어가는 작업이라는 걸 다시 한 번 느꼈음.

그래서 구조를 안정해둔 상태에서 AI한테 다 만들어줘 하는건 다음과 같은 이유에서 비효율이 존재한다.

- 전체 그림의 랜덤성이 강해지고
- 그걸 이해하고 수정하는데 시간이 오래걸리고
- 내 목표를 달성하는 결과물로 수정하려면 어차피 많이 잘라내고 수정해야될 가능성이 매우 크다.

그래서 오늘 느낀건, 아무것도 모를 때 ideation은 AI의 도움을 받더라도, 전체 구조를 AI한테 잡아달라고 하는 것 보다는 내가 원하는 것을 해결할 수 있는 구조 방법에 대한건 인간이 좀 고민을 하더라도 고민하고 이를 잘 구조화해서 프롬프팅 하는게 시간이나 리소스 측면에서 더 빠르게 원하는 결과물을 얻어대는 방향성이 아닐까 싶다. 정해진 규칙 내에서 자율성을 발휘하도록 해야되는 것 같음

이거는 AI가 똑똑해지는 것과 관계없는 문제일 것 같음. 아무리 똑똑해져도 내가 뭘 원하는지 내 머리를 열어볼 수는 없으니까....

## 느낀 점2

이어서 만들면서 느낀 점..

GPT가 짜준 SKILL.md 를 보다보니, 어떤 파일들을 읽는 작업은 굳이 '이 파일에서 이러이러한 내용을 확인해서 처리해줘' 같이 프롬프르틀 써서 처리할 필요가 없다는 생각이 들었음.

왜냐하면 전체 파일에서 일부 정보만 필요한데, 파일에서 데이터를 읽어내는 작업을 프롬프트로 시키면 자연스럽게 해당 파일 내용을 전부 context로 가져오기 때문에 비효율이 발생하지 않을까 하는 생각..

그러면 어떻게 해야되는지? -> 해당 파일에서 읽어야하는 부분을 읽어오는 script를 짜고 agent가 script 실행결과만 읽도록 하는 방식이 context 아끼고 토큰 아끼는 길이라는 생각이 들었다.

그리고 혹시 해당 파일이 markdown이라서 code parsing이 불편하다면, 데이터의 특성을 고려해서 가능하다면 code로 parsing하기 상대적으로 수월한 json이나 csv 같은 구조화된 파일 형식으로 만드는 것도 세트상품으로 같이 고려해볼만한 부분이라는 점을 배웠음.
