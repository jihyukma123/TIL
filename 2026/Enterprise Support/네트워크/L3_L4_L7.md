# OSI 7 레이어 중, L3/L4/L7

ChatGPT Enterprise 관련 맥락에서는 이 3개의 계층에 대해서만 먼저 이해를 하면 된다고 한다.

알아보자.

## 왜 Header insertion을 위해서는 L7레이어를 보는 장비가 필요할까?

L7과 같은 구체적인 레이어에 대한 이야기를 하기 전에 어떤 상황에서 이런 점들이 고려되는지 보자.

상황은 다음과 같다.

`엔터프라이즈 네트워크 내부에서 직원이 브라우저로 chatgpt.com에 접속해서 대화를 시도한다.`

이 때 사용자의 요청이 ChatGPT 서버로 전송될텐데, 이는 그 과정에 대한 내용이다.

### Outbound: 브라우저 -> ChatGPT서버까지 요청이 내부에서 밖으로 전달되는 과정

L7(Application) -> L1(Physical) 순서로 한 계층씩 내려가면서 데이터를 추가하는 형식으로 패킷이 생성됨

근데 여기서 궁금증.

L7 -> L1은 그러면 일단 최초에는 사용자의 컴퓨터에서 다 발생하는 일인가?
-> 그렇다. 이 '캡슐화' 과정은 사용자 PC에서 발생함. 'HTTP 요청 만들고, TLS 암호화하고, TCP 헤더 붙이고, IP 헤더 붙이고, 전기 신호로 변환' OS의 네트워크 스택이 수행하는 일이라고 함.

하지만 패킷이 PC를 떠나서 목적지까지 가는 동안 거치는 경로 위의 모든 장비는 자기가 처리하는 레이어에서 껍데기를 벗겨서 필요한 정보를 읽고, 다시 씌워서 전달하는 작업을 수행한다고 함.

예를 들어서, 스위치라는 장치는 데이터링크 레이어인 L2까지만 본다고 한다. 왜냐하면 스위치는 '같은 네트워크 안에서 패킷을 올바르게 전달하는 것'이라는 역할을 수행하는 장치이다. 그렇기 때문에, 같은 네트워크 안에 있는 '장비'를 식별하기 위한 정보(MAC Address)만 필요하고, IP주소(네트워크 밖의 다른 네트워크 주소)는 알 필요가 없다.

MAC 주소는 하드웨어에 박혀있는 물리적인 식별자로, 제조 시 NIC(네트워크 카드)에 고정된 주소라고 함.

(구체적으로 전기신호인 패킷이 어떻게 스위치에서 인식되어서 L2의 MAC 주소가 해석되는지 알아보기 전에, 이것부터 알고 가자. 나는 왜 바로 L2에서 필요한 정보를 접근해서 가져오지 못할까 라는 의문이 생겼음. 객체를 생각해보면, packet.l2 같이 접근해서 바로 각각 레이어에서 필요한 정보만 가져가면 되지 않을까? 라는 생각을 했는데 애초에 동작하는 방식이 그렇지가 않았다. 패킷은 `[ L2 헤더 [ L3 헤더 [ L4 헤더 [ L7 데이터 ] ] ] ]`이런 식으로 nested 구조를 가지고 있어서, L3 헤더를 보려면 먼저 L2 헤더를 parsing하는 과정이 꼭 필요하다. 봉투안에 봉투가 있는 느낌이라, 안쪽 봉투를 보려면 일단 바깥쪽 봉투를 열어봐야되는 구조인거임. 다만, 물리적으로 보면 이런 nested 구조는 아니고, 메모리상에서는 평탄한 구조의 연속된 바이트열임. `[목적지MAC][출발지MAC][타입][IP헤더][TCP헤더][데이터]` 이런 식으로 일련의 데이터라는 뜻이다. 그렇겠지 그냥 01010101 이런 거일텐데...무튼 앞 레이어 정보를 읽어야 다음 레이어 정보가 어디서부터 시작하는지, 길이는 얼마인지을 알아서 그만큼의 비트를 읽어서 parsing 할 수 있음...와 이런거 보면 내가 평소에 대부분의 시간을 아무런 지장이나 딜레이를 못느끼면서 인터넷을 활용하는게 얼마나 대단한 기술인지 새삼 느끼게 됨. 그리고 왜 이런 논리적으로 연쇄해석이 필요한 구조로 만들었을까? 그냥 각각 정보를 독립적으로 고정해서 처리하면 안됐나?? -> 왜 이런 구조로 만들었을까? -> TBC.....)

1단계 — 전기신호 수신 (L1)

포트로 들어오는 전기신호(또는 광신호)를 비트열로 변환. 이건 PHY 칩이 담당.

2단계 — 프레임 경계 파악

비트열이 어디서 시작하고 끝나는지 구분. Ethernet 프레임은 시작을 알리는 **Preamble (7바이트) + SFD (1바이트)**가 앞에 붙어있다.

[ Preamble | SFD | 목적지MAC | 출발지MAC | 타입 | 데이터 | FCS ]
이걸 보고 "여기서부터 프레임 시작"을 인식.

3단계 — FCS 체크섬 검증

프레임 맨 끝 4바이트가 FCS (Frame Check Sequence). 전송 중 비트가 깨졌는지 확인. 깨져 있으면 그냥 버린다. 여기까지도 하드웨어가 처리.

4단계 — MAC 주소 읽기

FCS 통과하면 이제 앞에서 0~5번째 바이트 = 목적지 MAC, 6~11번째 바이트 = 출발지 MAC 읽기.

이게 전부다. 여기서 멈춤. IP 헤더는 건드리지도 않음.

5단계 — TCAM 조회

읽은 목적지 MAC을 TCAM에 던져서 포트 번호 반환받음. 이게 하드웨어 레벨에서 병렬로 동시에 비교되기 때문에 사실상 O(1).

그리고 파악한 port로 전달 -> 해당되는 장비로 데이터가 전송됨.

즉 전체 패킷을 읽을 필요가 없이, 해당 장비로 전송하기 위해서 필요한 정보를 읽기 위해서 필요한 작업과, 정보만 읽어서 처리하고 끝남.
