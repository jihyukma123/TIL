# AI와 일하는 방식의 변화 — 개발 현장에서 먼저 나타난 트렌드

## 발표 개요

| 항목   | 내용                                                      |
| ------ | --------------------------------------------------------- |
| 주제   | AI와 일하는 방식의 변화 — 개발 현장에서 먼저 나타난 트렌드 |
| 대상   | IT 조직 구성원 (비개발자 포함)                            |
| 시간   | 30~40분 + Q&A                                             |
| 발표일 | 2026-04-13 (월)                                           |

## 발표 본문

---

### 도입 (5분)

- 지난 2~3년 동안 AI는 "질문에 답하는 도구"에서 "일을 맡길 수 있는 도구"로 빠르게 바뀌고 있다
- 이 변화는 개발 분야에서 가장 먼저, 가장 선명하게 나타났다
  - 코드는 텍스트라 AI가 바로 다룰 수 있고
  - 실행과 테스트로 결과를 비교적 빠르게 검증할 수 있고
  - 개발자는 불편하면 도구 자체를 바꿔버리는 집단이기 때문이다
- 그래서 개발 현장을 보면, 다른 직무에서도 앞으로 어떤 변화가 올지 조금 더 먼저 볼 수 있다

발표에서 전하고 싶은 핵심은 하나다.

> AI 활용의 중심이 "좋은 답을 받는 것"에서 "일을 잘 맡기고, 그 일이 안정적으로 굴러가게 만드는 것"으로 이동하고 있다.

---

### 1부 — 변화의 큰 흐름 (5분)

개발 현장에서 사람과 AI의 관계는 대략 네 단계로 변해왔다.

| 단계 | 사람과 AI의 관계 | 실제로는 어떤 모습이었나 | 대표 예시 |
| ---- | ---------------- | ------------------------ | --------- |
| 1단계 | AI에게 질문하고 내가 가져다 씀 | 코드나 오류를 복사해 질문하고, 답을 사람이 다시 반영 | ChatGPT |
| 2단계 | AI가 내 작업 맥락 안으로 들어옴 | IDE 안에서 파일을 읽고, 수정 제안을 하거나 직접 고침 | Copilot, Cursor |
| 3단계 | AI에게 작업 단위를 맡김 | "이 기능 만들어줘", "이 버그 잡아줘"처럼 결과 단위로 지시 | Claude Code, Codex 등 |
| 4단계 | AI가 잘 일하게 환경을 설계함 | 규칙, 검증, 도구, 권한 범위를 설계해 결과 편차를 줄임 | 각종 Agent 운영 방식 |

여기서 중요한 것은 도구 이름이 아니다.

- 사람이 AI를 어떻게 쓰느냐
- 어디까지를 직접 하고, 어디부터를 위임하느냐
- 결과를 좋게 만드는 핵심이 질문인지, 맥락인지, 운영 환경인지

이 포인트가 계속 바뀌고 있다는 점이 핵심이다.

---

### 2부 — 왜 다음 단계로 넘어갔나 (10분)

각 단계는 "더 좋아져서" 넘어간 것이기도 하지만, 더 정확히 말하면 이전 방식의 병목이 분명해졌기 때문에 다음 단계로 넘어갔다.

#### 1단계: 질문 중심 활용

초기에는 AI에게 질문하고 답을 받아오는 방식이 중심이었다.

- 빠른 설명, 요약, 예시 생성에는 유용했다
- 하지만 실무에서는 매번 맥락을 사람이 직접 설명해야 했다
- 코드, 파일 구조, 현재 상황, 원하는 결과를 계속 복사해서 붙여넣어야 했다

병목은 명확했다.

> "AI가 똑똑하지 않아서"가 아니라, AI가 내 일의 맥락을 모르기 때문에 매번 처음부터 다시 시작해야 했다.

그래서 나온 요구가 이것이다.

> AI가 내 작업 맥락 안에 바로 들어와 있으면 좋겠다.

#### 2단계: 맥락 통합

여기서 "맥락"은 단순한 참고 정보가 아니라, AI가 지금 무슨 일을 해야 하는지 이해하는 데 필요한 배경정보 전체를 뜻한다.

- 지금 보고 있는 파일이나 문서
- 이 작업의 목적
- 이미 정해진 규칙이나 정책
- 이전에 어떤 시도가 있었고 무엇이 문제였는지
- 최종적으로 어떤 결과가 맞는지에 대한 기준

즉, 맥락이란 "정답 그 자체"가 아니라, AI가 덜 헤매고 더 적절하게 판단하도록 도와주는 작업 배경이다.

그다음에는 AI가 IDE나 작업 도구 안으로 들어왔다.

- 더 이상 매번 코드를 복사해 붙여넣지 않아도 됐다
- AI가 현재 파일, 주변 코드, 프로젝트 구조를 보고 도와줄 수 있게 됐다
- 생산성은 확실히 좋아졌지만, 새로운 한계가 생겼다

그 한계는 이것이다.

> 여전히 사람이 하나하나 시켜야 했다.

- "이 함수 고쳐줘"
- "테스트 만들어줘"
- "이 부분 설명해줘"

작은 작업에는 좋았지만, 큰 작업은 사람이 계속 쪼개서 관리해야 했다.

그래서 나온 요구가 이것이다.

> 내가 세부 단계가 아니라 목표만 주면, AI가 중간 단계를 알아서 처리해주면 좋겠다.

#### 3단계: 자율 실행

최근에는 AI에게 더 큰 작업 단위를 맡기는 흐름이 강해졌다.

- 사람이 세부 구현을 하나씩 지시하지 않고
- 목표, 제약, 완료 조건을 주면
- AI가 여러 파일을 보고 수정하고, 필요한 중간 단계를 스스로 수행한다

이 단계에서 병목은 다시 바뀌었다.

> 이제 문제는 "시킬 수 있느냐"가 아니라 "안정적으로 같은 수준의 결과를 내느냐"가 됐다.

- 같은 지시를 해도 결과가 들쭉날쭉할 수 있고
- 엉뚱한 범위를 건드리거나
- 중간 판단이 어긋나면 전체 작업이 흔들릴 수 있다

즉, 자율성은 생겼지만 완결성과 일관성이 새로운 문제로 떠올랐다.

#### 4단계: 운영 환경 설계

그래서 최근에는 관심이 프롬프트 문구 자체보다, AI가 일하는 환경을 어떻게 설계할 것인가로 이동하고 있다.

여기서 말하는 환경은 대략 이런 것들이다.

- 어떤 문서를 항상 참고하게 할지
- 어떤 파일은 수정 가능하고, 어떤 파일은 건드리면 안 되는지
- 중간에 어떤 검증을 통과해야 다음 단계로 가는지
- 실패했을 때 어떻게 되돌리고 다시 시도할지

비개발자 기준으로 풀어 말하면 이렇다.

> 예전에는 "AI에게 뭐라고 말할까"가 중요했다면, 지금은 "AI가 실수하지 않게 어떤 판 위에서 일하게 할까"가 더 중요해지고 있다.

---

### 3부 — 지금 두드러지는 트렌드 3가지 (15분)

#### 1) 사람의 역할이 "실행"에서 "관리와 판단"으로 이동하고 있다

- 개발자의 일이 사라진다기보다, 일의 무게중심이 이동하고 있다
- 직접 코드를 타이핑하는 시간보다
- 무엇을 맡길지 정하고
- 결과가 맞는지 판단하고
- 잘못되면 어떤 피드백 구조를 줄지 설계하는 비중이 커지고 있다

즉, 사람의 역할이 점점 아래에서 손을 움직이는 쪽보다, 위에서 방향과 기준을 잡는 쪽으로 올라가고 있다.

Microsoft는 `Work Trend Index: The Frontier Firm` (2025)에서 AI 활용이 어시스턴트 → 디지털 동료 → 워크플로우 운영자 방향으로 진화한다고 정리했다.

Anthropic `2026 Agentic Coding Trends Report`도 비슷한 포인트를 짚는다. 소프트웨어 엔지니어링은 점점 "코드를 직접 쓰는 일"만이 아니라, 코드를 만드는 에이전트를 조율하고 결과를 평가하며 방향을 잡는 일로 이동하고 있다는 것이다.

#### 2) AI가 맡을 수 있는 작업의 범위가 커지고 있다

- 초기에 AI는 짧은 질문응답이나 보조 작업에 강했다
- 지금은 더 긴 흐름의 작업, 여러 단계가 연결된 작업까지 맡기는 시도가 늘고 있다
- 핵심은 단순히 "한 번에 더 많이 한다"가 아니라, 목표를 받으면 중간 단계를 이어서 처리하려는 방향으로 진화하고 있다는 점이다

Anthropic의 `2026 Agentic Coding Trends Report`는 대규모 코드베이스에서 에이전트가 비교적 긴 시간 동안 자율적으로 작업한 사례들을 소개한다. 숫자 자체보다 중요한 것은, AI 활용 단위가 "질문 1개"에서 "작업 흐름 1개"로 바뀌고 있다는 점이다.

#### 3) 그래서 경쟁력의 초점이 "질문 잘하기"에서 "운영 구조 설계"로 이동하고 있다

- 한때는 프롬프트를 잘 쓰는 사람이 앞서갔다
- 그다음에는 맥락을 얼마나 잘 붙여주느냐가 중요했다
- 지금은 반복적으로 좋은 결과를 내는 운영 방식을 만드는 쪽이 더 중요해지고 있다

예를 들어:

- 참고 문서와 정책을 어떻게 정리할 것인가
- 업무를 어떤 단위로 나눠 맡길 것인가
- 중간 검토는 어디서 할 것인가
- 실패했을 때 복구 루프를 어떻게 만들 것인가

이건 개발자만의 기술 문제가 아니라, 사실상 업무 설계 문제에 가깝다.

Deloitte `2026 Global Human Capital Trends`도 비슷한 메시지를 준다. AI를 기존 업무 위에 그냥 얹는 것보다, 사람과 AI의 상호작용 방식 자체를 다시 설계한 조직에서 더 큰 성과가 나온다는 것이다.

### 4부 — 왜 비개발자에게도 중요한가 (5분)

이 발표는 개발 사례를 중심으로 설명했지만, 이 흐름은 개발자만의 이야기가 아니다.

- 대부분의 지식 노동은 결국 문서, 판단, 검토, 의사소통, 반복 처리로 이루어진다
- 생성형 AI는 바로 이런 텍스트 기반 작업과 절차 기반 작업에 먼저 침투한다
- 그래서 개발에서 먼저 나타난 변화가, 기획, 운영, 법무, 마케팅, 리서치 같은 영역으로 확산될 가능성이 높다

중요한 것은 직무명이 아니라 패턴이다.

- 처음에는 AI에게 질문한다
- 그다음에는 자료와 맥락을 붙여준다
- 그다음에는 특정 작업을 맡긴다
- 결국에는 AI가 잘 일하게 하는 기준과 절차를 설계하게 된다

즉, 개발은 특수한 예외가 아니라 조금 먼저 도착한 선행 사례에 가깝다.

---

### 5부 — 정리 (5분)

오늘 이야기의 요지는 세 가지로 정리할 수 있다.

1. AI 활용은 "질문해서 답을 받는 것"에서 "일을 맡기고 운영하는 것"으로 이동하고 있다.
2. 개발은 그 변화가 가장 먼저, 가장 선명하게 나타난 분야다.
3. 앞으로의 차이는 모델 하나를 고르는 능력보다, AI가 안정적으로 일하도록 업무와 환경을 설계하는 능력에서 더 크게 벌어질 가능성이 높다.
