## 발표 본문

---

### 도입 (5분)

- 2022년 ChatGPT 출시 이후, 개발자들이 AI를 활용해 코딩하는 방식이 급격히 변해왔다
- 개발 분야에서 이 변화가 가장 빠르게 일어났다
  - 코드는 텍스트 기반이라 AI가 바로 다룰 수 있고, 실행하면 맞는지 틀린지 즉시 확인할 수 있고, 개발자는 불편하면 도구를 직접 만드는 사람들이기 때문
- 이 과정을 보면, 사람이 AI와 함께 일하는 방식의 변화가 어떻게 진행되고 있는지 볼 수 있다

---

### 1부 — 어떻게 변해왔나 (5분)

개발자들이 AI를 활용하는 방식이 이렇게 변해왔다:

| 방식                            | 실제로 한 일                                                              | 대표 도구              |
| ------------------------------- | ------------------------------------------------------------------------- | ---------------------- |
| AI에게 질문하기 (2022~)         | 코드를 복사해서 ChatGPT에 붙여넣고, 답변을 받아서 다시 내 코드에 붙여넣음 | ChatGPT                |
| AI를 작업 환경에 통합 (2023~)   | AI가 내 코드를 직접 보고, 내 파일을 직접 수정해줌. 복붙이 사라짐          | Cursor, Copilot        |
| AI에게 자율 실행 위임 (2025~)   | AI가 여러 파일을 돌아다니며 스스로 작업을 수행함                          | Claude Code, Codex CLI |
| AI가 일하는 환경을 설계 (2026~) | AI가 통제된 환경에서 일관된 결과를 만들도록 환경 자체를 설계함            | 위 도구 + 환경 설계    |

---

### 1-2부 — 왜 이렇게 변해왔나 (10분)

각 단계에서 사람과 AI의 관계가 어떻게 달라졌는지, 그리고 왜 달라질 수밖에 없었는지.

**1단계: AI에게 물어보고, 내가 가져다 쓰는 방식 (ChatGPT)**

핵심 문제: **AI를 쓰려면 매번 채팅창에 관련 정보를 직접 입력해야 했다**

- 코드를 복사해서 붙여넣고, 프로젝트 구조를 설명하고, 상황을 매번 알려줘야 했다
- 대화가 길어지거나 새 세션을 열면 맥락이 사라져서 처음부터 다시 설명해야 했다
- 이 과정이 번거로워서, 결국 단순한 질문에만 쓰게 되고 복잡한 실무에는 적용하기 어려웠다

→ 사람들이 원한 것: AI가 내 프로젝트의 맥락을 이미 알고 있는 상태에서 바로 일해줬으면

**2단계: AI가 내 작업 환경 안에서 같이 일하는 방식 (Cursor/Copilot)**

전환 포인트: **내 작업 환경 밖에 있던 AI가, 내 작업 환경 안으로 들어왔다**

- 더 이상 채팅창에 코드를 복붙할 필요가 없어졌다
- AI가 내 프로젝트 파일을 직접 읽고, 코드를 직접 수정해준다
- 1단계의 핵심 문제였던 "매번 맥락을 설명해야 하는 번거로움"이 해소됐다

핵심 문제: **매번 사람이 하나하나 지시하고 승인해야 했다**

- "이 함수 고쳐줘", "이 테스트 만들어줘"를 매번 사람이 트리거해야 했다
- 큰 작업을 통째로 맡기기 어려웠다. 작은 단위로 쪼개서 하나씩 시켜야 했다

→ 사람들이 원한 것: "이 기능 만들어줘" 하면 알아서 전체를 처리해줬으면

**3단계: AI가 알아서 실행하는 방식 (Claude Code/Codex)**

전환 포인트: **사람이 하나하나 지시하지 않아도, AI가 스스로 여러 단계를 실행하게 됐다**

- 2단계의 핵심 문제였던 "매번 사람이 트리거해야 하는 번거로움"이 해소됐다
- "이 기능 만들어줘" 하면 AI가 파일을 탐색하고, 코드를 수정하고, 테스트까지 스스로 수행한다

핵심 문제: **같은 지시를 해도 매번 다른 결과가 나왔다**

- LLM은 근본적으로 확률적으로 동작한다. 같은 작업을 10번 시키면 10번 다 다른 결과가 나올 수 있다
- 엉뚱한 파일을 수정하거나, 의도와 다른 방향으로 폭주하기도 했다
- 자율성이 높아진 만큼, 잘못된 방향으로 갈 때의 피해도 커졌다

→ 이 문제가 "AI에게 뭘 시킬지"보다 "AI가 일하는 환경을 어떻게 만들지"로 관심이 전환되는 계기가 됐다

**4단계: AI가 일하는 환경을 설계하는 방식 (Harness Engineering) — 현재 진행 중**

**이 변화를 가속시킨 또 다른 요소: AI 모델 성능의 획기적인 개선**

- 초기 모델(Sonnet 3.5 등)은 성능이 부족해서, 프롬프트를 정교하게 다듬는 것이 중요했다
- 최신 모델(GPT-5.4, Opus 4.6 등)은 성능 자체는 충분히 좋아졌다
- 더 이상 모델 성능이 병목이 아니라, 구조를 잘 짜고 확률적으로 잘못되는 것을 방지하는 것이 핵심이 됐다
- 즉, "AI가 못 해서" 문제가 아니라 "AI가 할 수 있는데 환경이 안 갖춰져서" 문제가 되는 시대

업계에서는 이 변화를 Prompt Engineering → Context Engineering → Harness Engineering이라고 부른다.

### 2부 — 지금은 어디로 가고 있나 (20분)

**1) 사람이 개입하는 지점이 점점 올라가고 있다**

```
코드를 직접 짠다
 → IDE에서 AI가 제안하면 수락한다 (Copilot, Cursor 초기)
  → CLI/터미널에서 AI가 자율 실행하고 결과를 확인한다 (Claude Code, Codex)
   → 환경만 설계하고, AI가 알아서 돌아간다 (Harness Engineering)
```

**2) Harness Engineering이라는 흐름**

- "AI에게 뭘 시킬지"보다 "AI가 일하는 환경을 어떻게 만들지"가 결과를 좌우한다
- 통제된 환경에서 AI Agent가 일관된 결과를 만들고, 그 환경 자체를 지속적으로 업데이트하는 것이 핵심

* **OpenAI Codex 팀 사례 (2025.8 ~ 2026.1)**
  - 엔지니어 3명이 5개월간 약 100만 줄 규모의 프로덕션 소프트웨어를 만들었다
  - 이 중 사람이 직접 작성한 코드는 단 한 줄도 없다
  - 엔지니어들이 한 일은 코드 작성도, 코드 리뷰도 아니었다
  - AI 에이전트가 스스로 잘 동작할 수 있는 환경 — 규칙, 제약, 피드백 구조 — 을 설계하고 계속 개선하는 것이었다
  - 사람의 역할이 "실행"에서 "환경 설계"로 완전히 이동한 사례

**3) 기존 도구들도 이 방향으로 변하고 있다**

- Cursor 3.0 사례 (2026.4.2 출시)
  - IDE에서 출발했지만, "에이전트 창"이라는 상위 인터페이스를 도입
  - 병렬 에이전트 실행, 워크트리 격리, 클라우드 실행 지원
  - 코드 에디터가 아니라 "에이전트 관리 환경"으로 진화 중

### 3부 — 정리 (5분)

- AI와 일하는 방식은 "질문을 잘 하는 것" → "맥락을 잘 주는 것" → "환경을 잘 설계하는 것"으로 변해왔다
- 도구는 계속 바뀌고 있지만, 방향은 일관적이다: 사람의 개입 지점이 점점 위로 올라가고 있다
- 모델 성능이 충분해진 지금, 병목은 AI의 능력이 아니라 AI가 일하는 구조와 환경으로 이동하고 있다
- 이 변화는 개발 분야에서 가장 먼저 일어나고 있지만, 같은 패턴은 다른 업무 영역에서도 나타날 것으로 보인다

### 개인적으로 대응하는 방식

- 내 업무를 파악하고, AI에게 위임 가능한 일을 구분하는 것부터 시작
- 실행자보다 관리자처럼 사고 — "이 일을 어떻게 시킬까"를 먼저 고민
- 돌려보고 → 의도와 다르면 원인 파악 → 수정 후 재실행을 반복
- 이터레이션을 통해 점점 내 업무에 최적화된 에이전트로 개선
- 현재: 일정 관리, 반복 업무 등을 맡기는 개인 비서 에이전트를 구조 설계부터 계속 업데이트 중
