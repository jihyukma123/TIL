# test 1 최초 프롬프트

최초 프롬프트 작성 시, model 간 응답의 차이가 있는지 궁금해서 알아봄.

프로젝트 init을 위한 프롬프트를 작성해서 줬을 때, 어떻게 동작하는지 궁금해서 테스트 함

reasoning effort는 medium으로 비교함.

## 최초 입력 프롬프트

```markdown
# 프로젝트: pin (프로토타입 이름)

## 만들고 싶은 것
CLI 코딩 에이전트(Codex 등)를 사용할 때, 에이전트가 만들어준 중요한 출력이
새 대화에 밀려 시야에서 사라지는 불편함이 있다. 같은 세션의 컨텍스트는
유지하면서, 그 내용을 옆에 pin해두고 계속 참조할 수 있는 도구를 만들고 싶다.

내가 직접 매일 쓰면서 발전시킬 toy project로 시작한다. 작게 간다.

## 작업 방식
이 프로젝트는 내가 코드를 절대 작성하지 않고, 오직 프롬프팅을 통해서
너와 같은 AI 에이전트가 내 의도대로, 그리고 안전하고 좋은 소프트웨어를
지속 가능한 방식으로 개발해나갈 수 있는 방향성으로, 알아서 자신의 환경을
만들고 iterative하게 개선해나가게 할 것이다.

## 시작
무엇부터 할지 정하는 것 자체를 같이 하고 싶다. 코드부터 짜지 말고,
이 프로젝트가 뭐고 어디서부터 시작하면 좋을지 먼저 같이 의논해줘.
모호한 부분은 추측하지 말고 질문해도 된다.
```

위 프롬프트를 아무것도 없는 프로젝트 상태에서 각 모델별로 5번씩 실행해서 모델별 특징이 있는지 이해해보고자 하였음.

## gpt-5.3-codex

### 1st

- 코드 작성하기 전에, 문제 정의와 첫 번째 해야될 작업 범위를 설정하자고 제안함
- v0 작업 범위를 설정해줌
- 로컬에 저장할 기술 스택 지정함(JSON/SQLite)
- 이어서 작업하기 위한 질문을 몇 가지 함. 기능 범위와, 성공 기준 등
- 다음 턴에서 바로 산출물을 뽑아내겠다고 함. 

약간 좀 뭔가 빠르게 만들려고 eager한 느낌을 받았음.

간단한 몇 가지만 정해주면, 산출물을 빠르게 만들려고 하는 느낌으로 만들어 줌

### 2nd

- 자신이 이해한 프로덕트 정체성을 제시함(but very short)
- 바로 핵심 문제와, MVP범위를 잘라서 제안함
- 핵심 결정사항 5개도 같이 제안
- 다음 턴에서 바로 산출물 만들겠다고 제안함.

eager한 느낌은 여전히 유지.

### 3rd

- 프로젝트 핵심 한줄로 요약(very short)
- 문제 정의 및 MVP 설정 필요성 제안
- MVP 범위 잘라서 제안
- 몇 가지 질문 제시
- 다음 턴에서 바로 스펙 산출물 작성 제안
eager, short 정체성, MVP범위 고정, 질문

### 4th

지금까지 느낌을 보면, 

- 간단하게 요약해서 프로젝트 핵심 주제 추천
- MVP 범위 한 가지 제안
- 설계용 질문
- 산출물 생성 제안 

이런 위주로 질문하고 이번에도 딱 그 형식에 맞춰서 뽑아줌

그리고 마찬가지로 gpt-5.5와 비교했을 때 훨씬 더 짧게 플랜을 제시함.

5.5 버전이 대체적으로
- 문제정의/프로덕트 정체성 설정의 길이 및 뎁스
- MVP범위 설정 시 여러 옵션 제안
- 전체 프로젝트 설계
- 프로덕트 설계 뿐만 아니라, 에이전트가 일할 수 있는 환경에 대한 설계도 같이 제안하는 경우가 있었음(gpt-5.3은 아예 없음)

## gpt-5.5 응답

### 1st

- 일단 엄청 긴 답변이 옴.
- 프로젝트 방향부터 정하는 것 관련해서 길게 작성됨
  - 자신이 이해한 프로젝트 목표를 먼저 state 함
    - -> good
  - 프로젝트 범위를 너무 크게 잡지 말라고 함
  - 핵심 가치를 설정함
- Dont 를 설정해줌
  - 하지말아야 할 것 -> 기능 단위로 제시함. 너무 투머치로 하지 말라고
- 초기 제품 원칙 및 그에 기반한 산출물
- 몇 가지 질문(프로덕트 완성을 위한)
- 그리고 마찬가지로, 다음 턴에서 바로 초안을 설계하겠다고 함. (정해진 문서 format을 이미 만들어두고..)

역시 약간 좀 eager한 느낌.

### 2nd 

- 이해한 문제에 대해 서술
- 작게 시작하자 
  - 작업 범위를 추천함.
- 질문을 물어봄(작업 단위 설정을 위한)
  - 그리고 프로덕트에 대한
- 순서대로 작업 제안
- 최초로 산출물 만들어보자가 아닌, 방향성을 정해보자 라는 방식으로 제안함.

### 3rd

- plan 제안 길이가 길다.
  - gpt-5.3은 대체적으로 짧은데, gpt-5.5는 훨씬 더 planning에 강점이 부여되어 있음.
    - 이게 model자체의 특성인지 아니면 codex에서 model별로 부여한 system prompt의 차이인지도 확인 필요
- 마찬가지로, 바로 구현으로 뛰지 않고 제품 범위와 프로토타입을 결정해야된다고 함.
- 자신이 이해한 프로덕트와, 핵심 가치 제안
- 정해야되는 것에 대해서 제안
  - MVP를 바로 고정하지 않고, 프로덕트를 구현할 수 있는 여러 형태를 제안함
  - 그리고 그 중에서 본인이 봤을 때 괜찮아 보이는 MVP 방식을 제안함.
- 제안한 MVP 방식에 대해서, 보다 심도있는 계획 제시함
  - 기능
  - 저장소 구조
- 관련해서 설계 질문
- 작업 순서 제안(코드 쓰지말고, 다음 순서대로 해라)
- agent가 알아서 하는 환경을 만들기 위해서 필요한 부분을 어떻게 해야되는지 강조하는 부분을 제시함(다른 답변은 안그랬던 것 같은데 좋은 포인트!!!) 프롬프트에서 제시한 내용을 잊지 않았음.
- 하지만 비슷한 eagerness 이슈

### 4th

지금까지 발견한 경향성
- 길다 일단. 훨씬 더 plan을 통합적으로 세워서 제공하는 경향성이 확실히 강해짐
  - 궁금점: gpt 버전의 문제가 아니라, codex 버전과 general 버전의 차이일까?
  - codex 버전이 더이상 안나오고 있는데 조던님 말씀처럼 general 모델 개발로 연결되는건가?
- 프로덕트 계획도 훨씬 더 딥하게 짜주고, 작업 방식(Agent가 알아서 잘 하는 방식)도 훨씬 더 잘 만들어줬음.

---

전체적으로, 5.3-codex 대비해서 gpt-5.5가 훨씬 더 통합적이고 complete 한 계획 방향성을 제시해주시는 함.


근데 is this better? 라고 판단은 어떻게 해야되지?


## 궁금해서 gpt-5.4도 테스트 해봄

길이는 확실히 더 길게 만들어줌.

그리고 더 상세한 계획을 물어봄

근데 작업 방식에 대한 이야기는 전혀 없음..

그리고 eager 하고..

그리고 뭔가 5.3-codex랑 비슷한 수준으로 동작하는걸로 느껴짐.

전체적으로 좀

- 프로덕트 정의
- MVP 범위 결정을 위한 정보 수집
- 제안
- 결정을 위해서 필요한 내용 질문

이정도 범위의 작업은 잘 하고, eagerness는 상대적으로 덜한듯. 조금 더 계획을 확정하자는 제안을 하는 경우가 있었음.

여전히 작업 방식(Agent native repository)에 대해서는 여러 번 돌려도 전혀 언급하지 않음.

뭐가 차이일까?


## 기본 시스템 프롬프트 제거 테스트

한참 삽질을 했다..ㅎㅎ

codex같은 경우 local에 `.codex/config.toml` 파일에, `model_instructions_file`값에 특정 파일 경로를 명시하고 해당 파일에 프롬프트를 작성하면 기본 프롬프트가 override된다고 나와있었음.

그래서 대체해서 입력이 안되는걸 확인하려고, codex 설정에서 proxy server로 요청이 보내지도록 해서 codex 에서 사용자가 질의를 입력했을 때 전송되는 정보를 찍어보려고 했음. 근데 file로 저장해달라고 했는데 아무리 해도 안돼서 1시간 동안 삽질을 했는데 알고보니 private 폴더로 만들어서 ide에서 보이지 않는거였음...

ㅋㅋㅋㅋㅋ 하하하

무튼 그래서 확인해보니, 옵션 조절 여부에 따라 model 뒤에 따라붙는 instruction이 빈 값으로 입력되는걸 확인했음

그래서 이제 해봐야 하는 것은, 모델별 프롬프트를 제거하고 요청을 전송했을 때 마찬가지로 내가 위에서 관찰했던 모델별 차이가 보일지가 궁금해서 테스트 해보면 되겠다.

## 제거 후 테스트

여전히 모델별 특성은 비슷하게 나오고 있다.

- 5.3-codex -> plan 이 전체적으로 짧고, 덜 전체적이며, 정해진 plan을 제시하고, eager to jump to work
- 5.5 -> plan이 훨씬 더 구체적이고, 다양한 옵션을 제시하며, 프로덕트 자체뿐만 아니라 프로덕트를 만드는 ai native 환경에 대해서도 더 자주 언급하고, jump to work 도 있지만 상의해보자 는 식의 뉘앙스를 더 많이 제시함.

비록 하나의 프롬프트만 가지고 비교했지만 

그래도 유의미한 특성 차이를 볼 수 있었음.

계획을 5.5로 해보고,implementation을 5.3-codex등으로 하느 방향성이 맞지 않을까
