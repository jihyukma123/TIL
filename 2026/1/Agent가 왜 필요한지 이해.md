# Agent의 필요성을 이해하게 됨

사람들이 왜 단일 agent 기반 질의를 통한 코딩에서 agentic workflow를 구성해서 일을 처리하게 되는지 이해했음.

기능 구현을 하는데 점차 agentic workflow로 발전하게 됨.

- 1단계: cursor에 처리하고자 하는 작업을 입력하고, cursor가 처리해줌 -> 조금씩 수정요청해서 마무리
- 2단계: 일관되게 적용되어야 하는 사항들을 하나의 문서로 정리해서 context에서 참고하도록 추가
- 3단계: 구현을 한 이후에, code-review 역할을 수행할 skill을 하나 붙여서, 변경사항에 대해서 code review를 하라고 수동으로 지시함

여기쯤 오면...'기능 구현 할떄마다 이걸 다 수동으로 해? 어차피 똑같은 단계를 거칠텐데' 같은 생각이 든다.

그래서 해보고 싶어지는게,

구현할 기능 입력하면

구체화 - 명세서 생성 - 구현 - 코드리뷰 <-> 개선 loop -> (필요하면 human in the loop) -> commit/merge 까지 수행하는 Agentic workflow 를 굴려보면 어떨까 하는 생각이 들게 되었음.

아마 다들 비슷하게 동일하게 반복하는 작업을 던지고 싶은 것 같다

그리고 이렇게 던졌을 때 의도된대로 동작하게 하기 위해서 그때그떄 필요한 정보만 LLM이 input으로 주기 위해서 다양한 orchestration 등 context management 관련 방법론이 등장하는거고.
