# 기술이 아닌 문제에 집중해야함.

> 문제 정의가 먼저다.

AI가 핫해서 그런지, 다들 AI로 뭐라도 '개선', 혹은 '효율화/자동화'를 하려는 것 같다.

이런 경우도 많아보인다. '그거 있잖아 구글에서 만든 지피티인가 머시기, 우리도 뭐라도 좀 해봐. 성과낼만한거'...

그래서 '모르겠는데 일단 AI 도입해.'류의 작업이 많이 이루어지고 있는 것 같은데 이거는 근본적인 문제가 있음.

문제에서 출발해서, 문제를 해결하기 위해서 필요한 적절한 기술을 찾아야 하는데...반대로 기술을 어떻게든 우겨넣을만한 문제를 발견해보라고 함...아니 당연히 있겠지 근데 기술에 우겨넣어서 풀어내면 그게 본질적인 문제를 풀겠냐고..사람들이 그걸 쓰겠냐고....

반대가 되어야 함.

`문제`에서 출발해야 한다.

'업무 자동화 프로세스 하나 만들어보자'가 아니라, 문제 발굴부터 똑바로 해야됨.

그리고 문제를 분석해서 어떤 방식으로 풀어내야하는지에 대해서 이해가 필요하다.

- 누가 쓰는지?
- 어디서 쓰는지?
- 어떻게 쓰는지?
- 왜 쓰는지?

이런 과정들을 거쳐서

1. 문제가 무엇인지
2. 문제 관련된 정보

를 파악한 후에야 어떤 기술로 주어진 문제를 주어진 조건 하에서 풀어낼지 정해볼 수 있음.

일단 모르겠고 AI 진행시켜는 제발 멈춰....이왕 하는거 잘 해봐야지ㅣ...

# 근래 경험과 연결해서...

내가 겪고 있는 문제가 있었음.

퇴근하고 공부해야된다는 생각에 책상 앞에 앉았는데 아니...글쎄 뭘 해야될지 잘 모르겠는거야. 해야될건 많다고 생각하는데 당장 오늘 지금 이 순간에 내가 뭘 공부해야될지 몰라서 시간을 허비하는 경우가 많았음.

그래서 이 문제를 풀려고, 처음에는 서비스를 만들려고 했음...

사용자가 사전에 입력한 공부 주제와, 최근 학습내역을 기반으로 그날 공부할 내용을 추천해주는 서비스를 웹으로 만들어보려고 했음.

근데 갑자기 Gemini CLI를 조금씩 사용해보면서, '이거 굳이 웹으로 만들어야 하나..?' 라는 생각이 들었음.

그래서 local에서 Gemini CLI를 써서 GEMINI.md 파일에 instruction을 써서 동작하는 Agent 형식으로 만들어도 충분히 내가 원하는 바를 성취할 수 있겠다는 생각이 들었음.

근데 만들다보니, 뭔가 이상함. 뭔가 구멍이 있어 서비스에.

그래서 이걸로는 내 문제가 잘 해결되지 않으리라는 것을 바로 깨달았다. 추천받은 주제만 공부해서는 내가 원하는 결과인 '성장'으로 이어질 가능성이 작아보였음.

그래서 다시 문제 분석에 딥다이브 했음.

그래서 왜 내가 책상 앞에 앉아있긴 한데 뭘 해야될지 결정못하고 허송세월을 보내는 이유가 무엇인지에 대해서 생각해본 결과,

- 목표가 명확하지 않음
- 왜 공부를 해야되는지가 명확하지 않음.

그래서 헤매고 있다고 생각했고, 이거는 서비스나 Agent를 활용해서 이 문제 자체를 풀어낼 수는 없다고 판단했음.

만약에 처음부터 문제에 집중해서 뭐가 진짜 문제인지 분석하는 시간을 가졌더라면, 웹서비스 만들겠답시고 낭비하지 않았겠지 시간을?
